The PriceCalculator is not well designed as it uses a series of if statements to handle different customer types. This makes the class difficult to extend since adding a new customer type requires modifying the existing method instead of adding new behavior in a separate place. A well-designed class should be open to add new behavior without changing existing code. As the number of pricing rules increase, the method will become larger, harder to read, and prone to more errors. 

This current design also mixes all discount logic into a single class, grouping the calculator and every customer type together. This violates good object-oriented design. A better approach would be separating discount behaviors into different classes so that each pricing rule can vary independently, improving maintainability, testing, and adaptability.
